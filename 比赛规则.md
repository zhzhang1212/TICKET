# 第一届百块奖金 Web 大赛
## 技术赛道赛题
题目名称
多态空间调度与高并发活动综合平台
一、 题目背景与核心目标
在高校日常生活中，学生获取与预订校园资源（如空教室、会议室、体育场馆、热门活动
门票）往往面临入口分散、规则复杂、高峰期系统崩溃等痛点。同时，在 AI 辅助编程
（如 Claude Code, Codex）普及的今天，快速生成基础的增删改查（CRUD）页面已不再
是开发者的核心壁垒。
核心目标：本题目要求参赛团队设计并实现一个“智约校园” Web 平台。作品不仅需要在
产品端提供清晰、易用、符合现代 Web 标准的交互体验，更需要在系统底层应对非标准
化业务规则、时空维度冲突以及高并发“秒杀”场景，全面考验参赛队伍的产品洞察力、系
统架构设计能力与 DevOps 自动化工程能力。
二、 核心业务场景与功能要求
平台需包含以下三大核心模块，参赛团队需在前后端完整实现功能闭环。
模块 A：多态空间建模与预订系统（核心架构挑战）
系统必须在底层原生统一两种物理和时间特性截然不同的校园资源，严禁使用扁平化的单
表应对所有场景。
1. 学术空间（会议室/讨论室）—— 连续时间与缓冲挑战
o 空间约束：原子化资源，不可分割。
o 时间约束：支持自定义时间段预订。系统必须在每次有效预订的起始前和
结束后，自动锁定各 5 分钟的缓冲期。该缓冲期对用户预订时长与额度计
算不可见，但在系统底层必须精准拦截任何与之重叠的并发预订。
2. 体育设施
o 空间约束：组合化资源。是多个场地预定举办比赛，还是单个场地预定。
o
时间约束：严格按 一个小时为离散槽位（Slot）开放。模块 B：校园热门活动聚合（高并发挑战）
平台需支持校园活动的发布与门票/名额抢注。

秒杀场景：模拟热门活动（如名人讲座、十佳歌手决赛）放票瞬间的流量洪峰。
架构要求：严禁将瞬间的读写流量直接压给关系型数据库。需通过引入缓存预热
（Redis）、消息队列（MQ）削峰及分布式锁等机制，确保系统在高并发下不宕
机、且绝对不出现“超发/超卖”现象。
模块 C：复杂状态机与动态规则引擎（逻辑解耦挑战）
1. 订单生命周期 (FSM)：
o 订单包含 待支付/确认、已确认、已取消、已爽约 等状态。
o 竞态条件防范：必须处理“幽灵支付”问题。例如用户在 15 分钟倒计时最
后一秒完成支付，而系统恰好触发超时取消逻辑，需利用数据库乐观锁
（Optimistic Locking）和延迟消息队列（Delayed MQ）保障状态绝对一
致。
2. 动态预约规则引擎：
o 废弃在核心代码中堆砌 if-else 的做法。针对多维规则（如：“本科生限
额 2 小时”、“信用分低于 80 额度减半”、“爽约扣减 10 分”），需实现责
任链模式（Chain of Responsibility）或表达式引擎，确保规则的可配置与
高扩展性。
三、 前端体验与产品质量标准
除了硬核的后端架构，前端工程同样需要遵循现代 Web 质量标准：
1. 响应式与交互设计：
o 页面结构清晰，重点信息层级分明，支持桌面端与移动端（Mobile First）
的平滑适配，确保在不同屏幕尺寸下均保持良好的可用性。
2. 无障碍设计 (Accessibility/a11y)：
o 参照 WCAG 2.2 标准，贯彻“可感知、可操作、可理解、健壮”四项原则。
要求图片具备替代文本（alt）、表单拥有明确标签、色彩对比度达标，并
支持核心流程的键盘导航。
3. 前端性能指标 (Core Web Vitals)：
o 关注页面加载与交互流畅度。参考当前 Web 核心体验指标：最大内容绘
制 (LCP) ≤ 2.5s，交互到下一次绘制 (INP) ≤ 200ms，累积布局偏移 (CLS)
≤ 0.1。
4. 基础 Web 安全：
o 参照 OWASP Top 10 框架，落实安全防范。避免明文暴露敏感信息，防
范 SQL 注入与 XSS 攻击，严格区分越权访问（BOLA/IDOR），并杜绝在
前端硬编码密钥配置。四、 现代化工程与 DevOps 要求 (CI/CD)
“在我的电脑上能跑”是无效的交付。本次比赛将 DevOps 自动化能力纳入核心考核：
1. 全量容器化 (Infrastructure as Code)：
o 提交的代码库必须包含多阶段构建的 Dockerfile 和编排文件 docker-
compose.yml。评委需能一键拉起包含前端、后端、数据库及缓存组件的
完整应用，严禁手动安装依赖。
2. 自动化流水线 (CI/CD)：
o 项目必须配置自动化流水线（如 GitHub Actions, GitLab CI）。所有的代码
提交必须能自动触发并跑通代码规范扫描（Linting）与自动化单元测试。
五、 提交内容要求
1. 作品演示链接：可公网访问的部署地址。
2. 项目工程仓库：包含完整源代码、README.md（写明架构设计、运行方式、测试
账号）。
3. 架构与产品说明文档：
o 重点阐述：多态空间数据库 Schema 设计、高并发防超卖策略、状态机流
转图、无障碍与性能优化实践。
4. 演示视频 (3-5 分钟)：展示核心功能流程与极端边界情况的拦截效果。
